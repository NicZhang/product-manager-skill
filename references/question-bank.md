# 问题库（Question Bank）

## 使用规则

- **非固定问卷** — 根据用户回答动态路由，不按顺序逐题询问
- **每阶段最多 3-6 个问题** — 聚焦最关键信息，避免疲劳轰炸
- **素材可替代问题** — 用户提供文档、截图、链接时，跳过已回答的问题
- **优先级：逻辑 > 功能 > 交互** — 逻辑不清楚时，功能描述毫无意义

---

## Phase 1：问题与背景（WHY）

### 开场问题（必问）

> "你想做什么？简单说说你的想法或遇到的问题。"

---

### 路径 A：模糊想法型

**触发条件**：用户没有具体问题，只是说"我想做 X"、"感觉可以做个 Y"

| 追问目标 | 问题示例 |
|---------|---------|
| 挖掘现有替代方案 | 现在没有这个东西，大家是怎么解决这个问题的？ |
| 识别真实痛点 | 你见过谁最需要这个？他们现在最头疼的是什么环节？ |
| 量化影响 | 如果一直没有这个，最坏的情况是什么？会影响多少人？ |
| 判断紧迫性 | 这件事现在不做，会有什么后果？是死不了但很烦，还是真的有损失？ |

---

### 路径 B：具体痛点型

**触发条件**：用户描述了真实遇到的问题或麻烦

| 追问目标 | 问题示例 |
|---------|---------|
| 明确时间线 | 这个问题大概什么时候开始出现的？是最近新出现的还是一直有？ |
| 了解当前应对方式 | 目前你们是怎么绕过这个问题的？每次要花多少时间或精力？ |
| 锁定理想结果 | 如果这个问题被解决了，你期待的状态是什么样的？ |

---

### 路径 C：带方案来型

**触发条件**：用户直接说"我要做一个功能/页面/系统"，跳过了问题描述

| 追问目标 | 问题示例 |
|---------|---------|
| 往回追溯根因 | 你说要做这个，是因为遇到了什么具体的麻烦吗？讲讲背景？ |
| 探索其他可能性 | 如果不做这个，还有没有别的方式能解决同样的问题？ |
| 找到核心诉求 | 如果只能保留这个方案最核心的一个价值，你觉得是什么？ |

---

### PM 挑战库

| 信号 | 挑战话术 |
|------|---------|
| "老板让做的" | 老板提这个，背后是在解决什么问题？他最担心的是什么？ |
| "竞品有这个功能" | 竞品做了，说明市场有需求。但你们的用户和竞品的用户是同一种人吗？他们遇到的场景一样吗？ |
| "所有人都需要" | 能不能描述一个最典型的用户？他昨天遇到这个问题的时候，具体在做什么？ |
| "越快越好" | 快到什么程度算快？如果下周上线但只有 50% 的功能，可以接受吗？ |

---

### Checkpoint 1 输出格式

```
✅ 问题确认

🎯 核心问题：[一句话描述]
😤 谁在痛：[受影响的人群]
📉 不解决的代价：[量化或描述后果]
⏰ 紧迫程度：[高 / 中 / 低，说明理由]
❌ 不是要解决的问题：[范围排除]

👉 下一步：进入 Phase 2，了解用户与场景
```

---

## Phase 2：用户与场景（WHO + WHERE）

### 开场问题（必问）

> "这个主要是谁在用？能描述一下他们的日常工作吗？"

---

### 用户画像不清

**触发条件**：用户说"就是普通用户"、"大家都能用"、描述过于笼统

| 追问目标 | 问题示例 |
|---------|---------|
| 锁定具体角色 | 你脑子里最典型的那个用户，他是什么职位？每天主要干什么？ |
| 区分新手与老手 | 这些用户第一次用这个产品时，和用了一年之后，需求会不同吗？ |
| 评估技术水平 | 这些用户平时用电脑习惯吗？他们用过类似的系统吗？ |

---

### 场景不清

**触发条件**：用户没有描述使用的具体情境、时间、地点

| 追问目标 | 问题示例 |
|---------|---------|
| 明确触发时机 | 用户什么时候会打开这个？是主动来的，还是被某个通知推来的？ |
| 了解前后文 | 用户在用这个之前刚做了什么？用完之后要去做什么？ |
| 判断协作模式 | 这个一般是一个人独立完成，还是需要多人配合？ |

---

### 多角色场景

**触发条件**：涉及多种用户类型（如操作员 + 管理员、客户 + 服务人员）

| 追问目标 | 问题示例 |
|---------|---------|
| 厘清角色关系 | 这几类人之间是什么关系？谁对谁负责？谁能看谁的数据？ |
| 权限边界 | 哪些操作只有特定角色能做？普通用户能做哪些，管理员才能做哪些？ |
| 数据流向 | 一个角色创建的内容，另一个角色什么时候能看到？需要审批吗？ |

---

### PM 挑战库

| 信号 | 挑战话术 |
|------|---------|
| "所有人都能用" | 如果一个 60 岁不太会用手机的人和一个天天用电脑的年轻人都要用，你会怎么设计？先聚焦哪个？ |
| "就是内部用" | 内部用户也有不同角色吧？比如运营、财务、技术，他们用的方式一样吗？ |
| "场景很简单" | 简单的场景通常意味着边界情况少。能举个例子说说一次完整的使用流程吗？ |

---

### Checkpoint 2 输出格式

```
✅ 用户与场景确认

👤 核心用户：[角色名称 + 一句话描述]
🔧 技术水平：[高 / 中 / 低]
📍 使用场景：[触发时机 + 使用环境]
🔄 使用流程：[用户使用前做什么 → 使用中 → 使用后做什么]
👥 涉及角色：[角色列表 + 权限关系]
❌ 不在范围的用户：[排除说明]

👉 下一步：进入 Phase 3，梳理方案与功能
```

## Phase 3：方案与功能（WHAT + HOW）

### 开场问题（必问）

> "你觉得这个产品必须有哪些功能才算能用？先列个清单。"

---

### 功能收集追问

| 追问目标 | 问题示例 |
|---------|---------|
| 挖掘可选功能 | 除了必须有的，还有哪些"有了更好但没有也能上线"的功能？ |
| 明确不做的事 | 有没有你明确不想做的功能，或者外面竞品有但你觉得不适合自己的？ |

---

### 逻辑追问（优先级最高）

**触发条件**：任何功能描述不清楚边界条件、异常情况或状态变化时，立即追问

| 追问目标 | 问题示例 |
|---------|---------|
| 前置条件 | 用户要使用这个功能，之前需要先完成什么操作或满足什么条件？ |
| 中断处理 | 用户操作到一半离开了，或者网络断了，这个操作应该怎么处理？ |
| 权限规则 | 不同角色对这个功能的操作权限有什么区别？有没有"只读"或"只能看自己的"这类限制？ |
| 数量与限制 | 这个功能有没有数量上限？比如最多能创建多少条，一次最多能操作多少个？ |
| 并发处理 | 两个人同时编辑同一条数据会怎样？先保存的生效，还是有锁的机制？ |
| 状态流转 | 这个数据/对象有哪些状态？每个状态之间怎么转换？哪些状态是单向的，不能回退的？ |
| 数据依赖 | 这个功能依赖哪些其他数据？如果依赖的数据被删了，这里会怎么显示？ |
| 计算规则 | 这里展示的数字是怎么算出来的？有没有四舍五入、截断、汇总的规则？ |

---

### 交互追问

| 追问目标 | 问题示例 |
|---------|---------|
| 高频操作 | 用户每天用这个最多的操作是什么？要特别顺手、快速？ |
| 数据量级 | 这个列表大概会有多少条数据？十几条还是几千条？需要分页或搜索吗？ |
| 操作反馈 | 用户做了这个操作之后，系统怎么告诉他成功了？失败了怎么提示？ |
| 批量操作 | 用户需要同时处理多条数据吗？比如批量删除、批量导出？ |
| 空状态 | 如果列表里没有数据，或者第一次使用，页面应该展示什么？ |

---

### MVP 砍需求

**触发条件**：功能列表超过 10 个、资源有限、上线时间紧时启用

| 追问目标 | 问题示例 |
|---------|---------|
| 时间逼迫 | 如果只有两周时间，只能做三个功能，你会选哪三个？ |
| 依赖识别 | 这些功能里有没有必须先做才能解锁其他功能的？先做最底层的那个。 |
| 功能合并 | 这两个功能解决的是同一个问题吗？能不能先做一个通用的，之后再细化？ |

---

### PM 挑战库

| 信号 | 挑战话术 |
|------|---------|
| 功能超过 10 个 | 这里有 12 个功能，全做完大概要三个月。先问你：哪三个没有就不能上线？其他的先砍掉。 |
| 功能互相矛盾 | 你说用户可以自由修改，但又说修改后需要审批。这两个在 A 情况下会打架，怎么处理？ |
| 边界不清楚 | 这个功能"方便用户管理"——管理是指查看、编辑、还是删除？能不能举个具体操作的例子？ |
| 只描述方案不描述问题 | 你说要做一个 AI 推荐，这个推荐是为了解决用户找不到什么？推荐的依据是什么？ |
| 逻辑含糊 | 你说"状态变了之后通知用户"，什么状态变化会触发通知？通知发到哪里？谁来通知？ |
| 缺非功能性需求 | 这个功能对响应速度有要求吗？如果加载超过 3 秒用户会怎样？数据要保留多久？ |

---

### Checkpoint 3 输出格式

```
✅ 方案与功能确认

📦 功能清单（MoSCoW）
  Must have（没有不能上线）：
    - [功能 1]
    - [功能 2]
  Should have（应该有）：
    - [功能 3]
  Could have（有了更好）：
    - [功能 4]
  Won't have（本次不做）：
    - [功能 5]

📐 核心逻辑规则
  - [规则 1：状态流转/权限/计算等]
  - [规则 2]

🔄 状态流转图（如适用）
  [状态 A] → 触发条件 → [状态 B] → 触发条件 → [状态 C]

❓ 待确认问题
  - [逻辑 1 还没有答案]
  - [逻辑 2 需要技术评估]

👉 下一步：整理成 PRD 草稿
```

## 素材处理问题库

### 文件确认（用户上传了文档）

| 追问目标 | 问题示例 |
|---------|---------|
| 核实提取的信息 | 我从这个文件里提取到了几个关键点，你确认一下理解对了吗？[列出要点] |
| 区分现状与历史 | 这个文档里的描述，是现在的情况，还是以前的方案？ |

---

### 截图确认（用户上传了截图）

| 追问目标 | 问题示例 |
|---------|---------|
| 理解截图意图 | 这张截图是你想要的效果，还是你想改掉的现状？ |
| 明确修改方向 | 截图里有没有你觉得不好的地方，想在新版本里改掉的？ |

---

### URL 确认（用户发了链接）

| 追问目标 | 问题示例 |
|---------|---------|
| 明确参考维度 | 这个链接里你最想借鉴的是什么？是它的功能、交互方式，还是界面风格？ |

---

### 矛盾处理（素材与口头描述冲突）

| 场景 | 处理话术 |
|------|---------|
| 文件说 A，你刚才说 B | 你刚才说的和文件里写的有点不一样，以哪个为准？还是说情况已经变了？ |
| 两份文件互相矛盾 | 这两个文件的说法不一样，哪个是最新的？还是两种情况都存在？ |
